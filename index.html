What is Odoo?

The Technology Stack

Understanding what powers Odoo helps you troubleshoot issues and communicate with developers.

Layer Technology What It Does
Database PostgreSQL (psql) Stores all your data - customers, orders, products, everything. Each Odoo database is a separate PostgreSQL database.
Backend Python 3 The brain of Odoo. All business logic, calculations, workflows, and data processing happen in Python.
Frontend JavaScript (OWL Framework) What you see in the browser. OWL (Odoo Web Library) is Odoo's own reactive framework, similar to React or Vue.
Web Server Werkzeug (Python) Handles HTTP requests. Odoo runs its own web server - no Apache or Nginx required (though often used as proxy).
API XML-RPC / JSON-RPC How external systems talk to Odoo. Used for integrations, mobile apps, and automation scripts.
Templating QWeb (XML) Odoo's template engine for reports, website pages, and email templates. Similar to Jinja2 but XML-based.

Types of Models

Three Model Types in Odoo

Odoo has three types of models. Each type stores data differently:

Technical Name What It Means Data Storage Examples
Model Regular model - normal screens Saved permanently in database Contacts, Sales Orders, Invoices, Products
TransientModel Temporary model - used for wizards (popup dialogs) Auto-deleted after 1 hour (by default) "Create Invoice" popup, "Register Payment" popup
AbstractModel Template model - provides features to other models Not stored (no database table) mail.thread (adds chatter to any model)

TransientModel = Wizards

A TransientModel (also called a "wizard") is a popup window that appears when you click action buttons. It collects input and performs an action.

"Transient" means temporary - the data you enter in these popups is automatically deleted after about an hour. This keeps the database clean.

An AbstractModel is a reusable set of features that can be added to any model. It doesn't create its own database table - it just provides functionality that other models can use.

How to Find a Model Name

  3. Go to any form view

How Data is Actually Stored - The Relational Database

Beyond the Excel Analogy

While the Excel analogy helps understand the basics, Odoo uses a relational database (PostgreSQL) which is more powerful. The key difference: tables are connected to each other using IDs.

When you create a Sales Order in Odoo, you're not just creating one record - you're creating multiple records across several tables, all linked together by their IDs.

  • order_id = 108 links back to header
  • product_id links to the product

Why IDs Matter - The Magic of Relational Databases

Every record in Odoo has a unique id (also called primary key). This ID is:

  • Auto-generated: Odoo creates it automatically when you save
  • Unique: No two records in the same table have the same ID
  • Permanent: Once assigned, the ID never changes
  • The Link: Other tables reference this ID to create relationships

Example: When Odoo displays "Acme Corp" on the sales order, it's actually reading partner_id = 42, then looking up id 42 in the res_partner table to get the name.

4. Field Storage - store=True vs store=False

Why This Matters

Whether a field is "stored" determines:

  • Can you filter by it?
  • Can you group reports by it?
  • How fast is it to read?
  • Does it take database space?

The Two Storage Options

store=True (Stored Field)

  • Has a column in the database
  • Value is saved permanently
  • Fast to read (direct lookup)
  • Can be used in Group By
  • Can be filtered and searched
  • Takes database space

Default for: Char, Integer, Boolean, Many2one, etc.

store=False (Computed On-the-fly)

  • No database column
  • Calculated every time you view it
  • Always shows latest value
  • Cannot use in Group By
  • Limited filtering options
  • No database space used

Default for: Computed fields, One2many, Related fields

Analogy: Calculator vs Notebook

store=True store=False
Like writing the answer in a notebook Like using a calculator each time
Answer is always ready to read Must calculate every time
Might be outdated if inputs changed Always fresh calculation
Takes paper (storage) No paper needed

Practical Impact

Example: Total Amount on Invoice

The field amount_total on an invoice:

  • If store=False: Calculated by summing all lines each time you view the invoice. Always accurate but cannot group invoices by total.
  • If store=True: Stored in database. Can group "Invoices by Total Amount" in reports. Must be recalculated when lines change.

Odoo's Choice: amount_total is stored (store=True) so you can use it in reports!

The Group By Limitation

If you try to "Group By" a non-stored field in reports or list views, you'll usually get an error or no results. This happens because:

  • Group By uses SQL, which needs a real database column
  • Non-stored fields don't have database columns
  • Odoo can't GROUP BY something that doesn't exist in the table

Exception: Some fields may appear as "Stored: False" in Developer Mode but still work for Group By. This happens when Odoo uses SQL translation tricks (like _field_to_sql) to redirect queries to an underlying stored field. See the Case Study below for a real example.

Common Frustration: "Why can't I group by this field?"

  • A computed field without store=True
  • A related field without store=True
  • A One2many field (never works for grouping)

Solution: Ask a developer to make the field stored, or group by a different field.

  • Current company: Product price varies by company

  • Data was imported/modified bypassing normal Odoo flows
  • A bug prevents recalculation

Quick fix: Edit and save the record to trigger recalculation. For bulk fixes, ask a developer to run a recomputation.

  • This can slow down the page significantly

7. Why Group By Requires Stored Fields

The Problem

You want to create a report: "Sales by Customer Country"

You try to group by a related field partner_id.country_id but it doesn't work. Why?

How Group By Works (Simple Explanation)

What Happens When You "Group By" in Odoo

  1. Odoo sends a request to the database (SQL query)
  2. The database groups records by the specified field
  3. The database returns grouped totals

The Problem: The database can only group by columns that exist in the database.

Non-stored fields don't have a database column, so the database doesn't know how to group by them!

Analogy: Library Card Catalog

Imagine you want to organize library books:

  • Stored field: Like having "Genre" written on each book's card. Easy to sort by genre.
  • Non-stored field: Like calculating "Days since published" - you'd have to look at each book individually. Can't pre-sort by it.

Solution: Make the Field Stored

When You Need to Group By a Computed/Related Field:

Ask your developer to add store=True to the field definition.

Trade-off: The field will take database space and might need recalculation triggers.

10. Domain Filters - Controlling Data

What Is a Domain?

A domain is a filter condition that limits which records are shown or selectable.

Domains are written as a list of conditions: [('field', 'operator', 'value')]

A Day Without Domains

Imagine you're setting up a Sales Order. When you click the Customer dropdown, you see EVERY contact in the system - 50,000 records including vendors, shipping addresses, employees, and archived contacts. Good luck finding your customer!

This is what Odoo would be like without domains. Every relational field would show every record, every search would return everything, and security rules wouldn't exist.

Domains are the unsung heroes that make Odoo actually usable - they filter what appears in dropdowns, power your saved searches, and enforce who can see what data.

How Domains Work

Domain Structure

A domain is a list of conditions (called "leaves") combined with logical operators.

Each condition is a tuple with 3 parts:

  1. Field name - The field to filter on (can use dot notation for related fields)
  2. Operator - How to compare (=, !=, in, like, etc.)
  3. Value - What to compare against

The "any" and "not any" Operators

These operators filter through relational fields with complex conditions:

The any operator:

  • Filter records where at least one related record matches a subdomain
  • Syntax: [('relation_field', 'any', [subdomain])]

The not any operator:

  • Filter records where no related records match a subdomain
  • Syntax: [('relation_field', 'not any', [subdomain])]

Examples:

# Find orders with NO delivered lines

# Find events with employee registrations
[('registration_ids', 'any', [('partner_id.employee', '=', True)])]

Note: The traditional dot notation [('order_line.product_id.type', '=', 'service')] is equivalent to using any and still works.

The "=?" Conditional Operator

The =? operator is special - it only applies the filter if the value is truthy (not False/None/empty):

# This is equivalent to:
# If company_id is set: [('company_id', '=', company_id)]
# If company_id is False/empty: [] (no filter, show all)

Combining Conditions

AND, OR, NOT Logic (Prefix Notation)

Understanding Prefix Notation

Key rule: The operator applies to the conditions that follow it.

Simple OR:

['|', ('state', '=', 'draft'), ('state', '=', 'sent')]

â†’ state is draft OR state is sent

  â†“    â†“       â†“              â†“              â†“

â†’ First | applies to (result of second |) and C
â†’ Second | applies to A and B
â†’ Final: A OR B OR C

  â†“       â†“          â†“       â†“              â†“

# Easier alternative:

5. Complex: (A AND B) OR (C AND D):

Incremental Mass Edit (List View Feature)

What is Incremental Mass Edit?

A built-in list view feature that allows you to update multiple records at once using mathematical operators. Instead of setting a fixed value, you can modify existing values incrementally.

Available in: Any list view where inline editing is enabled (Odoo 15+)

Key benefit: Update prices, quantities, or amounts across hundreds of records with a single edit

Supported Operators

Operator Meaning Example Result (if current = 100)
+= Add to current value +=10 110
-= Subtract from current value -=25 75
*= Multiply current value *=1.15 115 (15% increase)
/= Divide current value /=2 50

How It Works Internally

When you use an operator like +=10, Odoo:

  1. Parses the input - The parsers.js module detects operator patterns (e.g., /^[+\-*/]\s*=\s*(-?\d+)/)
  2. Creates an Operation object - Stores the operator and operand separately
  3. Shows confirmation - Displays "Update to: Sales Price + 10" (not a computed result)
  4. Computes per record - For each selected record, applies: operation.compute(currentValue)
  5. Calls web_save_multi - Saves each record with its individually calculated value

Key insight: The operation is computed per record, so each record gets its own calculated value based on its current data.

Field Types That Support Operators

The operator tip only appears for numeric field types:

  • Integer - Quantities, counts, sequence numbers
  • Float - Measurements, percentages, ratings
  • Monetary - Prices, costs, amounts

For other field types (like Selection, Many2one, Char), you can only set fixed values.

Example: 10% Price Increase Across Products

Scenario: Annual price revision - increase all product sales prices by 10%.

Steps:

  1. Go to Sales > Products > Products
  2. Use filters/search to find products to update (or select all)
  3. Click the checkbox to select records (or use "Select all X")
  4. Click on the Sales Price column of any selected record
  5. Enter: *=1.10
  6. Press Tab or click away - confirmation dialog appears
  7. Review: "Field: Sales Price, Update to: Sales Price * 1.10"
  8. Click Update to apply to all selected records

Result: Each product's price is multiplied by 1.10 (10% increase). A $100 product becomes $110, a $50 product becomes $55.

Scenario: Warehouse manager needs to increase minimum stock levels by 5 units for all products.

  2. Select the products to update
  3. Edit the Min Quantity (or reordering rule minimum) column

Important Considerations

  • Domain selection limitation: If you click "Select all X records" (domain selection), only records on the current page are actually updated
  • Validation: Records that fail validation (e.g., negative price not allowed) are skipped, and you'll see "X of Y records are valid"
  • Precision: Division may produce decimal results - be aware of rounding for Integer fields
  • No undo: Mass edits cannot be easily undone - consider testing on a few records first
  • Access rights: Users need write permission on the field being edited

Technical Implementation (For Developers)

The feature is implemented in the web module:

  • Parser: addons/web/static/src/views/fields/parsers.js - parseOperation() function
  • Operation class: addons/web/static/src/model/relational_model/operation.js
  • Confirmation dialog: addons/web/static/src/views/list/list_confirmation_dialog.js
  • Multi-save logic: addons/web/static/src/model/relational_model/dynamic_list.js - multiSave()
  • Backend: addons/web/models/models.py - web_save_multi() method

The Operation class simply stores operator and operand, then computes: value + operand, value - operand, value * operand, or value / operand

Use Server Actions when:

  • Business rule should apply consistently to all records
  • Workflow needs to progress automatically

15. Integration Methods: Connecting Odoo with External Systems

Four Ways to Exchange Data with Odoo

Odoo provides multiple methods to integrate with external systems. Each serves different needs:

  • Webhooks (Incoming) - External systems call Odoo via simple HTTP
  • Webhooks (Outgoing) - Odoo sends data to external systems via HTTP POST
  • JSON/2 API - Modern RESTful API for external applications (Odoo 19+)
  • XML-RPC / JSON-RPC - Legacy RPC protocols (deprecated in Odoo 19)

This section focuses on practical, non-technical explanations so functional consultants can understand when to use each method.

Incoming Webhooks: External Systems â†’ Odoo

What Are Incoming Webhooks?

Incoming webhooks allow external systems to send data INTO Odoo by making a simple HTTP request to a special URL.

Real-world analogy: Like a mailbox where external systems can drop messages for Odoo to process.

Location: Settings > Technical > Automation > Automated Actions (Trigger: On webhook)

How to Set Up an Incoming Webhook

Example: Receive Orders from E-commerce Website

Scenario: Your Shopify store needs to create sales orders in Odoo when customers purchase online.

Steps:

  1. Go to Settings > Technical > Automation > Automated Actions
  2. Click Create
  3. Set Automation Rule Name: "Receive Shopify Orders"
  4. Set Model: Sale Order (sale.order)
  5. Set Trigger: On webhook
  6. Copy the Secret URL generated (looks like: https://yourcompany.odoo.com/web/hook/abc123-def456)
  7. In Target Record field, write code to parse incoming data: model.create({'partner_id': payload.get('customer_id'), 'amount_total': payload.get('total')})
  8. Save and share the URL with your Shopify developer

What happens: When Shopify sends order data to that URL, Odoo automatically creates a sale order.

How Incoming Webhooks Work

  • URL Format: /web/hook/{unique-uuid}
  • Method: Accepts both GET and POST requests
  • Authentication: Public endpoint (no login required) - Security through URL secrecy
  • Data Format: Receives JSON or form data, accessible via payload variable
  • Processing: Automated action executes immediately when request arrives
  • Response: Returns JSON: {"status": "ok"} or {"status": "error"}

Incoming Webhooks Best Practices

  • Security: Keep the webhook URL secret - it acts as authentication
  • Regenerate URL: If compromised, use "Rotate Secret" button to get new URL
  • Validation: Always validate incoming data before creating records
  • Error handling: Use try/except in your code to handle invalid data gracefully
  • Logging: Enable "Log Calls" during testing to debug issues
  • Idempotency: Check if record already exists to avoid duplicates

Outgoing Webhooks: Odoo â†’ External Systems

What Are Outgoing Webhooks?

Outgoing webhooks allow Odoo to send data to external systems via HTTP POST when specific events happen.

Real-world analogy: Like Odoo sending a text message to another system saying "Hey, something happened!"

Location: Settings > Technical > Actions > Server Actions (Action Type: Send Webhook Notification)

Example: Notify Warehouse System of New Orders

Scenario: When sales order is confirmed in Odoo, automatically notify your warehouse management system to prepare shipment.

Steps:

  1. Create Automated Action on Sale Order model
  2. Set Trigger: State is set to
  3. Set Trigger Field: "sale" (confirmed)
  4. In Actions tab, add Server Action: Send Webhook Notification
  5. Set Webhook URL: https://warehouse.yourcompany.com/api/incoming-orders
  6. Select Fields to Send: Order Number, Customer Name, Order Lines, Delivery Address

What happens: Every time an order is confirmed, Odoo POSTs order data to your warehouse system.

How Outgoing Webhooks Work

  • Method: HTTP POST with JSON payload
  • Payload Structure: Always includes _id, _model, _name + your selected fields
  • Authentication: None by default - receiving system should implement their own auth
  • Retry: No automatic retry - if external system is down, data is lost
  • Timeout: Default HTTP timeout applies (typically 60 seconds)
  • Synchronous: Blocks the Odoo transaction until webhook completes

Outgoing Webhooks Best Practices

  • Fast endpoints: External system should respond quickly to avoid blocking Odoo
  • Error handling: External endpoint returning error will fail the Odoo transaction
  • Sensitive data: Don't send password fields or tokens - they bypass group restrictions
  • Testing: Use webhook testing sites like webhook.site during development
  • Consider async: For slow external systems, use scheduled action instead

JSON/2 API: Modern RESTful Integration (Odoo 19+)

What Is JSON/2 API?

JSON/2 is Odoo's modern, RESTful-style API for external applications to read and write Odoo data.

Real-world analogy: Like a librarian's desk where external applications can request books (data) or return them.

Key advantage: Uses standard Bearer token authentication instead of username/password on every call.

How to Use JSON/2 API

Example: Mobile App Reading Product Catalog

Scenario: Your mobile app needs to display Odoo products with current pricing.

API Call:

POST /json/2/product.product/search_read
Authorization: Bearer your-api-key-here
Content-Type: application/json

{
  "domain": [["sale_ok", "=", true]],
  "fields": ["name", "list_price", "image_128"]
}

Response: Returns array of products with requested fields

JSON/2 API Characteristics

  • Endpoint Format: /json/2/{model}/{method}
  • Authentication: Bearer token (API key) in Authorization header
  • Methods Available: All public Odoo model methods (search, read, write, create, unlink, etc.)
  • Request Format: JSON POST body with method parameters
  • Response Format: Clean JSON response (not wrapped in RPC protocol)
  • IDs Parameter: Pass record IDs as array in request body, not in URL

When to Use JSON/2 API

  • Building mobile applications that need Odoo data
  • Creating custom dashboards or reporting tools
  • Integrating with modern web applications
  • When you need stateless, token-based authentication
  • Any new integration project (this is the recommended approach)

XML-RPC & JSON-RPC: Legacy RPC APIs

Deprecation Notice

Important: XML-RPC (/xmlrpc, /xmlrpc/2) and JSON-RPC (/jsonrpc) are deprecated in Odoo 19 and will be removed in Odoo 20.

If you're maintaining existing integrations using these protocols, plan migration to JSON/2 API.

What Are RPC APIs?

RPC (Remote Procedure Call) APIs allow external applications to call Odoo functions as if they were local.

Authentication: Requires database name, user ID, and password on every call

Endpoints:

  • /xmlrpc/2/object - Model method calls (execute_kw)
  • /jsonrpc - Similar to XML-RPC but uses JSON format

Record Rules = Security filtering (WHO can see WHAT records)

  • Purpose: Filter/UX convenience
  • Applied: Only in specific context
  • Scope: Local - affects only that view/field
  • User control: Users can clear/modify filters
  • Performance: Evaluated only when needed
  • Visibility: Visible as filters

20. Common Mistakes & How to Avoid Them

Mistake 1: Using Non-Stored Fields in Group By

Wrong

"I want to group orders by customer country"

Uses: partner_id.country_id (related, not stored)

Result: Error or empty grouping

Correct

Create a stored related field:

partner_country_id = fields.Many2one(related='partner_id.country_id', store=True)

Result: Works perfectly

Mistake 2: Confusing Field Type and Widget

Wrong Thinking

"I need a radio button field"

Creates a new field type

Correct

"I need a Selection field displayed as radio buttons"

Uses: Selection field with widget="radio"

Mistake 3: Over-Using Many2many

Wrong

"Product can have multiple suppliers"

Uses: Many2many between product and supplier

Problem: Can't store price per supplier!

Correct

Create a "Product Supplier" model with:

  • Many2one to product
  • Many2one to supplier
  • Price, lead time, etc.

Mistake 4: Forgetting ondelete

Problem

You have orders linked to customers. If someone deletes a customer, what happens to their orders?

  • Default (set null): Orders remain but customer field becomes empty - data loss!
  • cascade: Orders get deleted too - dangerous!
  • restrict: Cannot delete customer with orders - safe!

Best Practice: Always specify ondelete='restrict' for important business relationships.

Mistake 5: Not Using Tracking

Best Practice

For important fields (status, assigned user, amounts), add tracking=True

This creates an audit trail in the chatter showing who changed what and when.

24. Context - The Hidden Parameter

The Mystery of the Pre-Filled Form

You click "Create Invoice" from a Sales Order, and the form already has the customer, products, and amounts filled in. Magic? No - it's context.

You navigate to Contacts from a Sales Team and only see contacts for that team. How does it know? Context.

Context is the invisible messenger that carries information between screens in Odoo. Understanding it helps you troubleshoot "why is this pre-filled?" and "why can't I see my record?"

What is Context?

Context is invisible information that travels with every action in Odoo. It tells views and actions how to behave, what defaults to set, and how to filter data. Think of it as instructions whispered behind the scenes.

Example: Sales Team Button

When you click "Quotations" from a Sales Team, the context includes:

25. Constraints - Data Validation Rules

What Are Constraints?

Constraints are validation rules that prevent bad data from being saved. When you see an error like "Barcode must be unique" during a product import, that's a constraint blocking duplicate barcodes before they cause inventory problems.

Why Constraints Exist

Without constraints, users could:

  • Enter negative values where only positive makes sense (like negative stock quantities)
  • Create duplicate records that should be unique (like two products with the same barcode)
  • Create circular relationships (Department A reports to B, B reports to A)
  • Enter invalid percentages (like 150% discount)

Constraints catch these issues at save time and display a clear error message explaining what's wrong.

Two Types of Constraints

SQL Constraints

What: Rules enforced by the database itself

When checked: When data is written to database

Speed: Very fast (database-level)

Best for:

  • Uniqueness (no duplicates)
  • Simple value checks (positive numbers)
  • Required combinations

Example errors you'll see:

"Code must be unique."

"Quotation validity days cannot be negative."

Python Constraints

What: Custom validation logic in code

When checked: Before data reaches database

Flexibility: Can check anything (related records, complex logic)

Best for:

  • Business logic validation
  • Cross-field checks
  • Relationship validation

Example errors you'll see:

"You cannot create recursive departments."

"Prepayment percentage must be a valid percentage."

SQL vs Python Constraints: Decision Matrix

When developers need to add validation rules, they must choose between SQL and Python constraints. Here's how to decide:

Pros and Cons Comparison

SQL Constraints

Pros:

  • Extremely fast (database-native)
  • 100% reliable - can't be bypassed even by raw SQL
  • Works even when ORM is bypassed
  • Less code to maintain
  • Enforced at the lowest level

Cons:

  • Limited to single-table checks
  • Can't access related records
  • Error messages can be cryptic
  • No access to Python logic or context
  • Harder to test in isolation

Python Constraints

Pros:

  • Full flexibility - any logic possible
  • Access to ORM and related records
  • Clear, customizable error messages
  • Can use context and environment
  • Easy to debug and test

Cons:

  • Slower (runs Python code)
  • Can be bypassed with raw SQL
  • More code to maintain
  • Only runs through ORM operations
  • Must handle recordsets properly

The Golden Rule for Developers

Use SQL constraints when possible, Python constraints when necessary.

SQL constraints are simpler, faster, and more reliable. Only reach for Python when you need:

  • To check data in other tables/models
  • Complex business logic that SQL can't express
  • Context-aware validation
  • Rich, dynamic error messages

Python Constraints (@api.constrains)

These use the @api.constrains decorator and can contain complex business logic:

Example 1: Preventing Circular References

Model: HR Department

@api.constrains('parent_id')
def _check_parent_id(self):
    if self._has_cycle():
        raise ValidationError('You cannot create recursive departments.')

What it prevents: Department A â†’ Parent: B â†’ Parent: A (circular loop)

Why Python: Checking for cycles requires traversing relationships - can't be done in SQL

Example 2: Percentage Validation

Model: Company Settings (prepayment)

@api.constrains('prepayment_percent')
def _check_prepayment_percent(self):
    for company in self:
        if company.portal_confirmation_pay and not (0 < company.prepayment_percent <= 1.0):
            raise ValidationError('Prepayment percentage must be a valid percentage.')

What it prevents: Invalid percentages like 0%, -10%, or 150%

Why Python: The check depends on another field (portal_confirmation_pay)

Example 3: Cross-Company Product Validation

Model: Sale Order

@api.constrains('company_id', 'order_line')
def _check_order_line_company_id(self):
    for order in self:
        companies = order.order_line.product_id.company_id
        if companies and companies != order.company_id:
            raise ValidationError(
                'Your quotation contains products from another company.'
            )

What it prevents: Adding Company B's products to Company A's quotation

Why Python: Requires checking related records across multiple models

How to Identify Constraint Errors

When you see a validation error popup in Odoo, it's usually from a constraint. Here's how to understand them:

SQL Constraint Error

Usually appears as:

  • "duplicate key value violates unique constraint"
  • "check constraint violated"
  • Technical message followed by the custom error message

Fix: The value you're entering already exists or doesn't meet the rule

Python Constraint Error

Usually appears as:

  • A clear message in a popup
  • "Validation Error" in the title
  • Human-readable explanation of the problem

Fix: Follow the instructions in the error message

For Consultants: What This Means

Key Takeaways

  • Constraints protect data quality: They prevent mistakes before they happen
  • Error messages are clues: Read them carefully - they tell you exactly what's wrong
  • Can't be bypassed: Unlike access rights, constraints apply to everyone including admins
  • Studio can't add them: Constraints require code - if you need custom validation, you need a developer

When Users Report Constraint Errors

  1. Read the error message - It usually explains the problem clearly
  2. Check for duplicates - Uniqueness constraints are the most common cause
  3. Look at related records - Python constraints often check relationships
  4. Verify the data makes sense - Is the percentage valid? Is the quantity positive?

Constraints vs Required Fields

Don't confuse these two concepts:

  • Required Field: Must have a value (any value) - enforced by required=True
  • Constraint: Must have a valid value that meets specific rules

A field can be both required AND constrained - for example, the field must exist AND be positive.
